# 泥人剧场项目改进建议

> 日期：2026-05-20
> 项目：niren-drama (泥人剧场)
> 技术栈：Spring Boot 3.2 + Java 17 + Vue 3 + TypeScript + MySQL 8.0

---

## 1. 后端架构师（关注性能/扩展性）

### 1.1 线程池配置需动态化

**现状**：`AsyncConfig.java` 中硬编码了线程池参数（corePoolSize=4, maxPoolSize=8）

**问题**：
- 无法根据运行时负载动态调整
- 高并发场景下可能成为瓶颈
- 不同环境（开发/测试/生产）需要不同配置

**建议**：
```java
@Configuration
public class AsyncConfig {

    @Value("${niren.async.ai.core-pool-size:4}")
    private int corePoolSize;

    @Value("${niren.async.ai.max-pool-size:8}")
    private int maxPoolSize;

    @Value("${niren.async.ai.queue-capacity:100}")
    private int queueCapacity;

    @Bean(name = "aiTaskExecutor")
    public Executor aiTaskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(corePoolSize);
        executor.setMaxPoolSize(maxPoolSize);
        executor.setQueueCapacity(queueCapacity);
        // 建议添加监控
        executor.setThreadNamePrefix("ai-task-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}
```

### 1.2 缺少缓存层

**现状**：Redis 已配置但未启用，频繁查询未缓存

**建议**：
- 启用 Redis 缓存，用于：
  - AI 配置缓存（`AiConfig`）
  - 用户会话信息
  - 热点数据（项目列表、角色列表）
- 实现缓存穿透/击穿/雪崩防护

### 1.3 异步任务缺少持久化状态机

**现状**：`TaskRecord` 状态管理分散

**建议**：
- 引入状态机模式管理任务生命周期
- 添加任务超时自动清理机制
- 实现任务重试策略（指数退避）

### 1.4 缺少服务降级机制

**现状**：AI 服务调用无熔断/降级

**建议**：
- 引入 Resilience4j 或 Sentinel
- 对外部 AI 服务调用添加：
  - 熔断器（CircuitBreaker）
  - 重试（Retry）
  - 限流（RateLimiter）
  - 超时控制（TimeLimiter）

### 1.5 模块化建议

**现状**：所有功能在单体应用中

**建议**（中长期）：
- 将 AI 服务调用抽取为独立微服务
- 视频合成服务独立部署
- 使用消息队列（RabbitMQ/Kafka）解耦长任务

## 2. 前端专家（关注组件复用/状态管理）

### 2.1 状态管理过于简单

**现状**：仅有一个 `user.ts` store，业务状态分散在各组件

**问题**：
- 组件间状态共享困难
- 数据一致性难以保证
- 组件刷新时状态丢失

**建议**：
```typescript
// stores/project.ts
export const useProjectStore = defineStore('project', () => {
  const currentProject = ref<Project | null>(null)
  const projectList = ref<Project[]>([])
  const loading = ref(false)

  async function fetchProjects() { ... }
  async function setCurrentProject(id: string) { ... }
})

// stores/ai-config.ts
export const useAiConfigStore = defineStore('aiConfig', () => {
  const configs = ref<AiConfig[]>([])
  const activeConfig = computed(() => ...)
  // ...
})
```

### 2.2 组件复用性不足

**现状**：多个 View 组件存在重复代码

**建议**：
- 抽取通用业务组件：
  - `ConfirmDialog` - 确认对话框
  - `LoadingOverlay` - 加载遮罩
  - `EmptyState` - 空状态展示
  - `ErrorBoundary` - 错误边界
  - `PaginationWrapper` - 分页封装

### 2.3 路由守卫逻辑可优化

**现状**：路由守卫中直接操作 DOM 跳转

**建议**：
```typescript
// router/index.ts
router.beforeEach((to, from, next) => {
  const userStore = useUserStore()

  if (to.meta.requiresAuth && !userStore.isLoggedIn) {
    next({ name: 'login', query: { redirect: to.fullPath } })
    return
  }

  if (to.meta.requiresAdmin && !userStore.isAdmin) {
    next({ name: 'dashboard' })
    return
  }

  next()
})
```

### 2.4 API 层类型安全不足

**现状**：API 返回值缺乏 TypeScript 类型定义

**建议**：
```typescript
// types/api.d.ts
interface ApiResponse<T> {
  code: number
  message: string
  data: T
}

interface PaginatedData<T> {
  records: T[]
  total: number
  current: number
  size: number
}

// api/project.ts
export function getProjectList(params: ProjectQueryParams): Promise<ApiResponse<PaginatedData<Project>>> {
  return request.get('/projects', { params })
}
```

### 2.5 缺少全局错误处理

**建议**：
- 创建 `useError` composable
- 统一处理网络错误、业务错误、权限错误
- 添加错误上报机制

## 3. 安全工程师（关注常见漏洞）

### 3.1 CORS 配置过于宽松

**现状**：`SecurityConfig.java` 允许所有来源

```java
config.setAllowedOriginPatterns(List.of("*")); // 危险！
```

**建议**：
```java
@Value("${niren.cors.allowed-origins:http://localhost:5173}")
private String allowedOrigins;

@Bean
public CorsConfigurationSource corsConfigurationSource() {
    CorsConfiguration config = new CorsConfiguration();
    config.setAllowedOrigins(Arrays.asList(allowedOrigins.split(",")));
    // ...
}
```

### 3.2 JWT Secret 硬编码

**现状**：`application.yml` 中存在默认 JWT Secret

**建议**：
- 强制要求通过环境变量设置
- 启动时检测是否为默认值并拒绝启动
- 定期轮换 Secret

### 3.3 SQL 注入风险

**现状**：`SchemaMigrationRunner` 使用字符串拼接 DDL

**建议**：
- 迁移脚本使用参数化查询
- 对用户输入严格校验
- 使用 MyBatis-Plus 的条件构造器避免 SQL 注入

### 3.4 文件上传安全

**现状**：上传路径和大小限制配置存在

**建议**：
- 添加文件类型白名单校验
- 文件名随机化处理
- 上传目录禁止执行权限
- 添加病毒扫描（可选）

### 3.5 敏感信息泄露

**现状**：
- `TencentCosConfig` 包含硬编码凭证
- 日志可能输出敏感信息

**建议**：
- 所有凭证通过环境变量注入
- 日志脱敏处理
- API 响应中移除敏感字段

### 3.6 缺少请求频率限制

**建议**：
- 对登录接口添加验证码或限流
- API 接口添加 Rate Limiting
- 使用 `@RateLimiter` 注解或网关限流

## 5. 数据库管理员（关注 SQL 效率）

### 5.1 缺少索引优化

**现状**：`SchemaMigrationRunner` 添加了部分索引，但可能不足

**建议**：
```sql
-- 常见查询场景的索引
CREATE INDEX idx_project_user_id ON drama_project(user_id);
CREATE INDEX idx_storyboard_project_id ON drama_storyboard(project_id);
CREATE INDEX idx_character_project_id ON drama_character(project_id);
CREATE INDEX idx_scene_project_id ON drama_scene(project_id);
CREATE INDEX idx_task_record_user_id ON drama_task_record(user_id, status);

-- 复合索引
CREATE INDEX idx_ai_config_user_type_default ON drama_ai_config(user_id, config_type, is_default);
```

### 5.2 大字段存储优化

**现状**：`LONGTEXT` 字段存储 JSON 数据

**建议**：
- 评估是否需要拆分表
- 考虑使用 JSON 列类型（MySQL 8.0 支持）
- 大字段考虑对象存储

### 5.3 缺少分页优化

**建议**：
- 大数据量表使用游标分页
- 避免 `SELECT COUNT(*)` 全表扫描
- 使用覆盖索引

### 5.4 事务管理

**建议**：
- 长事务拆分
- 避免在事务中调用外部服务
- 使用 `@Transactional(readOnly=true)` 优化只读查询

### 5.5 数据归档策略

**建议**：
- 历史任务记录定期归档
- 已完成项目软删除
- 日志表分区

### 5.6 连接池优化

**建议**：
```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20
      minimum-idle: 5
      idle-timeout: 300000
      max-lifetime: 1200000
      connection-timeout: 30000
```

## 6. 运维工程师（关注部署和可观测性）

### 6.1 缺少健康检查

**现状**：Docker Compose 中 MySQL/Redis 有健康检查，但后端没有

**建议**：
```yaml
backend:
  healthcheck:
    test: ["CMD", "curl", "-f", "http://localhost:8080/api/actuator/health"]
    interval: 30s
    timeout: 10s
    retries: 3
    start_period: 40s
```

添加 Spring Boot Actuator：
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
</dependency>
```

### 6.2 日志管理

**现状**：日志仅输出到文件

**建议**：
- 使用 ELK（Elasticsearch + Logstash + Kibana）或 Loki
- 结构化日志（JSON 格式）
- 日志级别动态调整
- 日志轮转和归档策略

### 6.3 监控告警

**建议**：
- 接入 Prometheus + Grafana
- 监控指标：
  - JVM 内存/GC
  - 线程池状态
  - API 响应时间
  - 错误率
  - AI 服务调用耗时
- 设置告警阈值

### 6.4 配置管理

**现状**：环境变量通过 docker-compose 传入

**建议**：
- 使用配置中心（Nacos、Apollo）
- 配置热更新
- 配置版本管理
- 敏感配置加密

### 6.5 CI/CD 流程

**建议**：
- 自动化构建流程
- 自动化测试（单元测试、集成测试）
- 蓝绿部署或金丝雀发布
- 回滚机制

### 6.6 备份策略

**建议**：
```yaml
# docker-compose.yml 添加备份服务
backup:
  image: mysql:8.0
  volumes:
    - ./backups:/backups
  command: >
    sh -c 'mysqldump -h mysql -u root -p$$MYSQL_ROOT_PASSWORD niren_drama | gzip > /backups/niren_$$(date +%Y%m%d_%H%M%S).sql.gz'
  environment:
    MYSQL_ROOT_PASSWORD: ${DB_PASSWORD:-NirenDrama2024!}
```

### 6.7 资源限制

**建议**：
```yaml
backend:
  deploy:
    resources:
      limits:
        cpus: '2'
        memory: 4G
      reservations:
        cpus: '1'
        memory: 2G
```
