# 全视频分镜主流程改造开发任务

## 1. 背景与目标

当前项目主流程中，分镜会先生成静态图片，再按动态镜头选择生成视频片段，最后合成成片。新的主流程调整为：

- 生图模型不再用于生成分镜图片。
- 每个分镜镜头都应生成视频，分镜产物以 `videoUrl` 为核心。
- 图片只用于角色资产、场景资产、风格参考、三角度参考等一致性辅助素材。
- 视频生成时应从角色/场景/参考资产中组装输入，供 ComfyUI 或其他视频模型保持人物、场景和风格一致。

本任务文档用于指导后续 AI/开发者按模块改造，不直接实现代码。

## 2. 现状流程冲突点

### 2.1 当前主流程

1. 生成剧本。
2. 生成分镜。
3. 调用 `VideoController.generateImages()` -> `StoryboardService.startGenerateStoryboardImages()` 为分镜生成图片。
4. 调用 `VideoController.generateDynamic()` -> `VideoCompositionService.startGenerateDynamicVideos()` 为选中的动态镜头生成视频。
5. 合成时 `VideoCompositionService.composeAsync()` 根据镜头是否有动态视频决定：
   - 有动态视频：下载 `videoUrl` 合成。
   - 没有动态视频：下载 `imageUrl` 做静态镜头微动合成。

### 2.2 与新流程冲突的点

- `Storyboard.imageUrl` 当前是分镜主产物，新流程下不应再作为每个镜头的必要产物。
- `dynamicSelected` 当前用于区分静态图镜头和动态视频镜头，新流程下所有分镜都应是视频，不应再作为是否生成视频的开关。
- `startCompose()` 当前校验“是否存在分镜图片”，新流程应改为校验“分镜视频是否齐备”。
- 前端仍有“生成图片”“静态图与动态片段（动态优先）”等主流程文案和入口。
- ComfyUI 视频 provider 当前只支持单图图生视频，不能接收角色图、场景图、三角度图等多参考图。
- 图片生成配置页目前更像“分镜图生成调试”，需要改成“资产/参考图生成调试”。

## 3. 后端改造任务

### 3.1 调整视频主流程接口语义

涉及文件：

- `backend/src/main/java/com/niren/drama/controller/VideoController.java`
- `backend/src/main/java/com/niren/drama/service/StoryboardService.java`
- `backend/src/main/java/com/niren/drama/service/VideoCompositionService.java`

任务：

1. 将 `/videos/generate-images/{projectId}` 从主流程中移除或降级为兼容接口。
2. 确认是否保留旧接口：
   - 推荐短期保留，但前端不再调用。
   - 文案改为“生成参考资产”时，应迁移到角色/场景/资产模块，不再写入分镜主图。
3. 将 `/videos/generate-dynamic/{projectId}` 的业务语义改为“生成分镜视频”。
4. `VideoCompositionService.startGenerateDynamicVideos()` 不再只筛选 `shouldUseDynamicVideo()`。
5. 默认生成项目下所有选中的分镜视频；如果请求传入 `shotIds`，只处理这些镜头。
6. 没有可生成分镜时，错误文案改为“项目下没有分镜数据，请先生成分镜”。
7. 任务类型可继续使用 `DYNAMIC_VIDEO_GEN`，但推荐后续改名为 `STORYBOARD_VIDEO_GEN` 或 `SHOT_VIDEO_GEN`。

验收标准：

- 前端点击“生成分镜视频”后，后端会为所有分镜发起视频生成。
- 不再因为 `dynamicSelected=false` 跳过镜头。
- 不再要求先有 `Storyboard.imageUrl` 才能进入视频主流程。

### 3.2 改造合成前置校验

涉及文件：

- `backend/src/main/java/com/niren/drama/service/VideoCompositionService.java`

任务：

1. `startCompose()` 当前使用 `imageUrl` 判断是否可合成，需要改为检查 `videoUrl`。
2. 合成前应校验参与合成的分镜是否全部存在 `videoUrl`。
3. 若仍要保留静态图 fallback，需要做成明确兼容逻辑，而不是主流程默认路径。
4. 错误文案从“分镜还没有生成图片，请先生成分镜图片”改为“分镜视频尚未生成完成，请先生成分镜视频”。
5. `composeAsync()` 中应优先使用 `videoUrl`，新流程可强制每个镜头走 `composeDynamicShot()`。
6. 如保留 `composeSingleShot()`，仅作为旧项目兼容或调试路径，并避免前端引导用户使用。

验收标准：

- 没有视频的分镜不能开始最终合成。
- 所有分镜都有视频后可以正常合成。
- 合成结果不再依赖分镜图片。

验收标准：

## 4. ComfyUI 多参考图改造任务

涉及文件：

- `backend/src/main/java/com/niren/drama/ai/impl/ComfyUiVideoProvider.java`
- `backend/src/main/java/com/niren/drama/ai/impl/ComfyUiWorkflowLoader.java`
- `backend/src/main/java/com/niren/drama/controller/AiConfigController.java`
- `frontend/src/views/settings/SettingsView.vue`
- `frontend/src/api/aiConfig.ts`

### 4.1 多图上传

任务：

1. 将 `ComfyUiVideoProvider.uploadImageIfRemote()` 扩展为可处理多张图片。
2. 每张远程图片都上传到 ComfyUI `/upload/image`。
3. 返回 ComfyUI 可识别的图片名列表。
4. 本地或已是 ComfyUI 文件名的图片直接透传。

验收标准：

- 输入 3 张远程参考图时，provider 会上传 3 次并拿到 3 个 ComfyUI 图片名。
- 任意一张上传失败时，错误信息能指出失败图片。

### 4.2 多 LoadImage 节点注入

任务：

1. `ComfyUiWorkflowLoader` 新增多图注入方法。
2. 推荐支持两种注入策略：
   - 按节点 ID 注入：最稳定。
   - 按 LoadImage 出现顺序注入：作为快速配置。
3. AI 配置 extra 示例：

```json
{
  "workflowFile": "user:your-video-workflow.json",
  "imageNodeIds": ["12", "37", "41"],
  "promptNodeId": "6"
}
```

4. 如果配置了 `imageNodeIds`，按节点 ID 注入。
5. 如果未配置 `imageNodeIds`，按 workflow 中 `LoadImage` 节点出现顺序注入。
6. 如果参考图数量多于节点数量，忽略多余图片并记录日志。
7. 如果参考图数量少于必需节点数量，应报错或允许空缺，具体由 extra 配置控制。

验收标准：

- 可将角色正面、角色侧面、场景图分别注入指定 ComfyUI 节点。
- 用户 workflow 中多个 `LoadImage` 节点不会全部被第一张图覆盖。
- 配置错误时返回清晰中文错误。

### 4.3 Prompt 节点注入增强

任务：

1. 当前 `injectPrompt()` 只修改第一个 `CLIPTextEncode` 节点。
2. 新流程建议支持 `promptNodeId`。
3. 如存在正向/负向 prompt，extra 可支持：

```json
{
  "positivePromptNodeId": "6",
  "negativePromptNodeId": "7"
}
```

4. 未配置节点 ID 时，保留旧的自动查找逻辑。

验收标准：

- 用户可以指定要注入的 prompt 节点。
- 不会误把正向 prompt 注入负向 prompt 节点。

### 4.4 配置页调试能力

任务：

1. AI 设置页的视频调试表单支持输入多张图片 URL。
2. 请求结构从 `imageUrl` 扩展为 `imageUrls`。
3. 后端新增或扩展 `/ai-configs/debug/generate-image-to-video` 支持多图。
4. 保留单图输入兼容。
5. 调试结果展示生成视频 URL 和实际注入节点信息。

验收标准：

- 配置页可以用 3 张参考图测试 ComfyUI 视频 workflow。
- 调试失败时能看到是 workflow、节点 ID、图片上传还是生成轮询失败。

4. 分镜生成视频时只读取当前分镜相关资产。

验收标准：

- 图片资产不再混在分镜主产物里。
- 后续可以扩展更多参考图类型。

## 8. 任务状态和文案改造

- 前端任务轮询和状态展示相关组件。

## 9. 推荐实施阶段

### 阶段一：前后端主流程改名与禁用分镜图片

目标：快速让主流程变成“分镜视频 -> 合成”。

任务：

1. 前端移除主流程中的生成分镜图片按钮和文案。
2. 后端 `startGenerateDynamicVideos()` 改为处理所有分镜。
3. 合成前置校验改为检查 `videoUrl`。
4. 保留旧图片接口但不再从主流程调用。

验收：用户可以不生成分镜图片，直接生成分镜视频并合成。

### 阶段二：角色/场景参考资产接入视频生成

目标：让视频生成可以使用一致性图片资产。

任务：

1. 梳理角色/场景/资产表字段。
2. 支持角色三角度图和场景参考图。
3. `AiVideoGenerationService` 组装多参考图上下文。
4. 记录每个分镜实际使用的参考图快照。

验收：同一角色在多个分镜视频中能使用同一组参考图。

### 阶段三：ComfyUI 多图 workflow 支持

目标：让 ComfyUI 视频模型接收多张图片。

任务：

1. 多图上传。
2. 多 LoadImage 节点注入。
3. prompt 节点 ID 配置。
4. 设置页多图调试。

验收：配置 3 张参考图和 3 个 LoadImage 节点后，ComfyUI 能正常生成视频。

### 阶段四：清理旧动态/静态混合概念

目标：统一产品语义。

任务：

1. 清理 `dynamicSelected` 在主流程中的判断。
2. 清理“静态图”“动态优先”“动态镜头”等旧文案。
3. 决定 `imageUrl` 是否作为旧数据兼容、封面图或废弃字段。
4. 增加必要测试。

验收：新项目全流程不再依赖静态分镜图片。

## 10. 测试清单

### 后端测试

1. 项目有 3 个分镜，未生成图片，调用生成分镜视频接口，应创建视频生成任务。
2. `dynamicSelected=false` 的分镜也应被生成视频。
3. 未生成任何视频时调用合成，应返回“请先生成分镜视频”。
4. 所有分镜都有 `videoUrl` 时调用合成，应进入合成任务。
5. ComfyUI 多图调试输入 3 张图，应上传并注入 3 个节点。
6. 配置错误的节点 ID，应返回明确错误。

### 前端测试

1. 创建页不再出现“生成分镜图片”主按钮。
2. 创建页提示为“生成分镜视频”。
3. 分镜页视频存在时显示视频预览。
4. 分镜页视频不存在时显示待生成状态。
5. 合成页在视频未齐备时禁用合成按钮。
6. 设置页可配置视频 workflow 的多图节点 ID。
7. 设置页可用多图调试生成视频。

### 兼容测试

1. 老项目只有 `imageUrl` 没有 `videoUrl` 时，前端应提示需要重新生成分镜视频。
2. 老接口 `/videos/generate-images/{projectId}` 如保留，不应影响新主流程。
3. 旧的单图 ComfyUI 调试仍可用。

## 11. AI 开发执行顺序建议

1. 先改 `VideoCompositionService.startGenerateDynamicVideos()`，让所有分镜都能生成视频。
2. 再改 `VideoCompositionService.startCompose()`，让合成依赖 `videoUrl`。
3. 再改 `frontend/src/api/video.ts` 和创建页/分镜页/合成页文案与按钮逻辑。
4. 再设计并添加数据库迁移字段。
5. 再扩展 `VideoAiProvider` 多图接口。
6. 再改 `ComfyUiVideoProvider` 和 `ComfyUiWorkflowLoader` 支持多图注入。
7. 最后改设置页多图调试和角色/场景资产绑定体验。

## 12. 不建议一次性做的事项

- 不建议立即删除 `imageUrl`、`imagePrompt`、`dynamicSelected` 字段，容易破坏旧数据。
- 不建议把所有图片 URL 都硬塞进 `Storyboard`，更推荐通过资产关系或 JSON 快照区分来源。
- 不建议继续把“动态视频”作为少数镜头功能，新流程应统一为“每个分镜都是视频镜头”。
- 不建议让 ComfyUI fallback 默认 workflow 承担多图视频生产，真实生产应优先要求用户配置 workflow 和节点 ID。
